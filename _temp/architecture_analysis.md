# 애드온 구조 분석 및 dodo 개선 계획서 (수정본)

본 문서는 RefineUI, LegionRemixHelper, EllesmereUI, Tukui, ElvUI 등 대표적인 WoW 애드온들의 아키텍처 구조를 분석하고, dodo 애드온의 불필요한 레거시 코드를 제거하여 EditMode 기반의 단일 아키텍처로 통합하는 최적화 방향을 제안합니다.

---

## 1. 대상 애드온 아키텍처 상세 분석

### [1] RefineUI
*   **설계 철학**: "Refine, don't replace." (블리자드 순정 UI를 완전히 대체하지 않고 세련되게 다듬는 방향)
*   **구조적 특징**:
    *   독자적인 프레임을 대량 생성하지 않고, 기존 블리자드 프레임의 위치, 크기, 텍스처를 후킹하여 가공.
    *   이벤트 핸들러를 순정 프레임과 공유하거나 가벼운 Mixin 구조 사용.
*   **장점**: 메모리 사용량이 극히 적고 UI 렌더링 오버헤드가 제로에 가까움.
*   **dodo 적용점**: 편의성/꾸미기 모듈 중 독자 프레임이 불필요한 경우(예: Minimap, Tooltip 등), 신규 프레임 생성을 지양하고 순정 프레임의 `HookScript` 및 `SetPoint` 수정 위주로 경량화.

### [2] LegionRemixHelper
*   **설계 철학**: 특정 인게임 이벤트 및 콘텐츠에 최적화된 독립형 QoL 유틸리티 모듈러.
*   **구조적 특징**:
    *   기능 단위로 모듈이 완벽히 분리되어 상호 의존성이 없음.
    *   모듈 내부에서 이벤트를 등록하되, 특정 지역에 진입하거나 옵션이 활성화될 때만 이벤트 등록(`RegisterEvent`)을 가동하고 비동작 상태일 때는 즉시 해제(`UnregisterEvent`).
*   **장점**: 백그라운드 CPU 점유율이 매우 낮음.
*   **dodo 적용점**: 퀘스트 자동화, 상인 편의기능, 낚시 찌 확대 등 실시간 전투와 관계없고 특정 상황에만 쓰이는 편의기능 모듈들은 평소에 이벤트를 100% 해제하여 CPU 점유 차단.

### [3] EllesmereUI
*   **설계 철학**: 하이엔드 쐐기(M+) 및 신화 레이더를 위한 극도로 단순화된 레이드 지향적 UI.
*   **구조적 특징**:
    *   oUF 유닛프레임 라이브러리를 가공하여 필요한 필수 프레임만 드로잉.
    *   WeakAuras와의 협동을 전제로 하여 애드온 자체 렌더링 로직을 극소화함.
    *   프레임 핫패스 내에서 임시 테이블 생성을 엄격히 통제하여 GC(Garbage Collection) 스파이크를 원천 예방.
*   **장점**: 프레임 레이트(FPS) 방어에 매우 탁월하며 전투 중 버벅임이 없음.
*   **dodo 적용점**: oUF 유닛프레임(`Unitframe.lua`, `ResourceBar.lua`) 업데이트 주기 최적화 및 불필요한 매 프레임 그라데이션 연산 최소화.

### [4] Tukui
*   **설계 철학**: 단일 글로벌 전역 객체를 사용하는 통합 미니멀리즘 프레임워크.
*   **구조적 특징**:
    *   전역 네임스페이스(`Tukui`)를 공유하며, 엔진부, 모듈부, 미디어부로 명확히 분리.
    *   외부 라이브러리(Ace3 등) 의존성을 배제하고 픽셀 퍼펙트 연산과 미디어를 엔진 내에 자체 구현.
*   **장점**: 로딩이 빠르고 일관된 비주얼을 제공하며 파일 크기 대비 기능 밀도가 높음.
*   **dodo 적용점**: dodo 전역 객체 내에 `Colors`, `Utils` 등의 공용 API를 중앙 집중화하고 모듈 파일들에서는 이를 캐싱하여 사용함으로써 중복 코드와 룩업 비용 제거.

### [5] ElvUI
*   **설계 철학**: 고도로 세분화된 대형 모듈러 프레임워크 및 커스터마이징의 극대화.
*   **구조적 특징**:
    *   **3분할 물리 폴더 구조**: `ElvUI`(엔진 및 모듈 메인), `ElvUI_Options`(설정 UI), `ElvUI_Libraries`(공유 라이브러리).
    *   **Load on Demand (LoD) 설정 패널**: 평상시/전투 중 절대 사용하지 않는 대형 설정창 코드를 완전히 분리된 하위 애드온(`ElvUI_Options`)으로 배치하여 게임 시작 시 로드하지 않음.
*   **장점**: 플레이 중 메모리 관리 효율 우수.
*   **dodo 적용점**: ElvUI의 설정 분리 철학을 dodo에 다르게 변형 적용. dodo는 별도의 설정창(OptionUI) 카테고리를 쓰지 않고 모든 설정을 `EditMode` 통합 설정 패널로 일원화하고 있으므로, 불필요해진 레거시 `OptionUI` 및 `Option.lua` 파일을 LoD로 분리할 필요 없이 **_temp 폴더로 이동하여 백업 보관**하고 로드 대상에서 제외함으로써 경량화 달성.

## 2. dodo 애드온의 현재 구조적 문제점 진단

1.  **미사용 레거시 설정창(Option UI) 상시 로딩**:
    *   현재 `dodo.toc`는 `UI/OptionUI.lua`, `UI/OptionUI.xml`, `Option.lua`를 항상 로딩하도록 설정되어 있음.
    *   인게임 설정 패널을 앞으로 사용하지 않고 모든 옵션을 `EditMode` 패널로 통합하여 관리할 예정임에도, 이 레거시 파일들이 매번 로딩되어 메모리 및 초기 구동 시간을 낭비함.
2.  **모듈 On/Off 제어부의 한계**:
    *   `Core/Modules.lua`에서 모든 모듈의 `OnEnable()`을 일괄 실행함.
    *   각 모듈이 자체 `OnEnable()` 내부에서 `dodo.DB.enableModuleName`을 체크하여 비활성화 처리를 하지만, 파일 로딩 및 메모리 할당, 일부 초기 설정 등록 비용은 이미 소모된 상태임.
3.  **지연 로딩 및 쓰로틀링 부족**:
    *   게임 접속 직후(`PLAYER_ENTERING_WORLD`) 수십 개의 QoL 모듈이 한꺼번에 초기화 연산을 수행하여 초기 접속 시 프레임 드랍 발생.

## 3. dodo 아키텍처 최적화 구체적 로드맵

### [Task 1] 레거시 Option UI 백업 및 설정 단일화
*   **목표**: 불필요한 파일 로딩 및 메모리 오버헤드 제거. 설정 인터페이스를 EditMode로 일원화.
*   **실행 계획**:
    1.  `dodo.toc` 파일에서 `UI/OptionUI.xml`, `UI/OptionUI.lua`, `Option.lua` 로드 항목 주석 처리 또는 삭제.
    2.  `UI/OptionUI.xml`, `UI/OptionUI.lua`, `Option.lua` 파일들을 `_temp` 폴더로 이동하여 안전하게 백업 보관.
    3.  모든 모듈의 세부 옵션은 `Editmode.lua` 설정 목록 및 각 모듈 내부의 `RegisterEditModeSetting` 동적 주입 API를 통해서만 관리되도록 일원화.

### [Task 2] 모듈 관리 커널 개선
*   **목표**: 사용하지 않는 모듈의 실행 비용을 완전히 0%로 통제.
*   **실행 계획**:
    1.  `Core/Modules.lua`의 `EnableModules` 루프 수정.
    2.  모듈을 실행(`module:OnEnable()`)하기 전, `dodo.DB` 내의 해당 모듈 활성화 마스터 스위치(`enableModuleName`)를 먼저 판별.
    3.  스위치가 `false`인 모듈은 `OnEnable()` 호출 자체를 건너뛰어 메모리 및 이벤트 할당 원천 봉쇄.

### [Task 3] 지연 초기화 및 핫패스 최적화
*   **목표**: 로그인 프레임 안정화 및 인게임 실시간 프레임 방어.
*   **실행 계획**:
    1.  비전투/QoL 모듈(예: Teleport, Friends, Merchant 등)은 즉시 로드하지 않고, `PLAYER_ENTERING_WORLD` 이벤트 발생 1~2초 후 `C_Timer.After`로 초기화를 순차적 지연 분산 실행.
    2.  체력바 등 핫패스 갱신 시 가비지 객체 생성을 차단하고, `C_CurveUtil` 등 무가비(Garbage-free) 순정 그라데이션 API 적극 활용.
